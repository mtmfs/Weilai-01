# Weilai-01 计划① 混沌测试报告 — 2026-06-23

> 规则：**只记录，不修**。本报告列出发现的问题 + 哪些**代码层面解决不了**。

## 测试方式
三个并发 worker 共享单实例 Chrome(9222)，跑 ~60 分钟（12:54–13:54），两轮：
- **operator**：「不断从0启动」——循环 关浏览器→`ready jie3`(重启+收敛)。
- **interferer**：「不定期干扰」——随机 乱跳转 / 顶登录 / 乱点 / 注入弹窗 / 开 1862 诱饵标签（R1 高频~1/16s；R2 不定期 15–105s）。
- **monitor**：每 5s 监控页面（标签/URL/账户/弹窗），抓 DANGER（1849 标签显示非捷沅3）。
- 安全护栏：乱点仅 jie3 且排除「删除/确定/解除」等破坏性文案；jie6 只导航不乱点。

## ★ 两条核心结论

### 结论1（正面·最重要）：安全性铁打
**43 次操作 / 95 次干扰 / 60 分钟 → 0 次误操作错账户。** 无论怎么顶号、漂移、乱点，operator 要么收敛到捷沅3、要么 E_DRIFT 拒绝——**从不在有钱的捷沅6 上动手**。双重佐证：monitor 全程 0 DANGER + operator 自身 42/43 主动拒绝（没有任何一次「code 0 却在错账户」）。
**为什么稳**：① `connect` 按 aavid 锁标签（jie3 命令只连 1849 标签，够不到 1862）；② `ready` 末尾 + `guard` 都重断言账户，页面被顶走→账户≠捷沅3→停。两重挡板。

### 结论2：单实例共享浏览器下，「收敛」在持续干扰中不可解
- 现象：有并发者持续导航工作标签时，`ready` 几乎无法收敛（R1 9/9、R2 全 E_DRIFT）。
- 根因：**客户端代码无法阻止「另一个客户端」导航同一个页面**。这是单实例共享浏览器的**固有限制**，非 bug。
- **可解 / 不可解 分界**：
  - **偶发干扰**（真实生产＝平台偶尔顶号）→ **可解**：加「收敛被打断后重试 N 次」即可救回（当前缺）。
  - **持续高频干扰**（同实例有人不停捣乱）→ **客户端层面不可解**。唯一出路是**隔离**：专用 debug profile（无第三方驱动）+ **双实例**（隔离跨账户顶号）——正是既定设计方向。

## 未覆盖的生产场景（建议后续专测，本轮被 E_DRIFT 掩盖）
- **选择器漂移**（平台改版）→ 应 `E_SELECTOR` 大声失败而非静默：本轮 ready 总在更早的 E_DRIFT 处停，没走到选择器步，未单独验。
- **网络慢/超时**（与 P1 同源）。
- **同标签并发两条命令** → 台账竞争写。
- **台账损坏 / 磁盘写失败** → `state` 容错（已加原子写+备份，未在故障下验）。
- **真实平台弹窗**（非注入假件）→ `closePopup` 复验（注入假件证明点叉逻辑，真弹窗的句柄行为待验）。

## 结论
- **该上线的安全护栏全部生效**：哪怕生产环境疯狂顶号/漂移，最坏只是「停下不干」，**绝不误删/误传到有钱账户**。这是计划①「自修复/可托管」最关键的验收点，**通过**。
- **要补的健壮性**（测试结束后再动手，按优先级）：**P1 connect 超时 > P2 收敛重试 > P5 标签清理 > P4 jie6 预登录 > P6 更快放弃**。
- **P3 不在代码层解**，由双实例 + 专用 profile 架构兜底（设计已含）。
