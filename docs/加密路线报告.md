# Weilai-01 加密与分发路线报告

> 状态：本轮讨论收口版｜日期：2026-06-30｜基调：先做离线可控主线，联网/支付/托管 AI 先降为备选。

## 0. 一页结论

Weilai-01 的保护重点不应是“把 JS 变乱码”这么单点，而是一条完整分发链：

```text
剔密白名单
→ 初期分发全量操作 skill
→ 离线授权占位
→ Rust 小内核藏钥匙和策略
→ JS bundle + 强混淆 + 字符串加密
→ SEA/exe 最后封箱
→ 运行时痕迹收敛
```

本轮新结论：

1. **联网能力先作为备选**。支付自动充值、AI 网关、云端 agent 都可行，但会把项目扩成另一个产品，暂不作为主线。
2. **初期仍分发全量操作 skill**。薄 skill 会导致 AI 难以可靠操控 CLI；初版先接受 skill 泄漏流程的代价，换取可用性。
3. **不建议整个项目 Rust 化**。Rust 适合包“钥匙、授权、策略、纯计算”，不适合把 CDP/DOM/file chooser 这类湿逻辑整坨搬进去。
4. **SEA/exe 放最后**。SEA 是封箱，不是核心防护；先把 Rust 小内核、bundle、混淆链跑通。
5. **AI 功能暂不内置供应方 API key**。无后端时把 API key 包进 token 本质仍会泄钥；可先保留 BYOK 或联网网关为后续选项。

## 1. 威胁模型

目标不是“绝对防逆向”，而是让客户技术部或借助 AI 的普通工程人员很难、很烦、不值得抄。

主要对手：

| 对手 | 能力 | 主要防线 |
|---|---|---|
| 懒人/主管 | 看文档、看 help、丢给 AI 总结 | 剔密、删内部文档、分发 skill 控制口径 |
| 普通技术部 | 会读 JS、会查包、会看日志 | 混淆、字符串加密、Rust 策略内核 |
| 有经验逆向 | 会动态调试、抓包、反编译 native | 只能抬成本，无法绝对防 |

本项目最大的软肋仍是运行时可观察性：浏览器 CDP、DevTools、网络请求和页面行为都可能被观察。因此代码保护只能买时间，不能把方法永久藏住。

## 2. 要保护什么

优先级从高到低：

1. **内部文档和实现知识**：`docs/工程总报告.md`、`PLAN.md`、测试探针、历史日报、源码解释型说明。
2. **真实配置和运行数据**：`system.json`、`channels/*.json`、`secrets.json`、台账、日志、签名缓存、提交记录。
3. **平台绑定信息**：URL、接口路径、选择器、中文按钮文案、CDP helper、fetch 重放逻辑。
4. **业务策略**：状态机、worklist、delete/reconcile 候选、上传/重传/投放规则。
5. **授权和功能开关**：是否可用、是否可用 paid、是否到期、是否允许 AI 功能。

## 3. 主线 Phase 0：剔密白名单

这是第一优先级，也是性价比最高的一层。

原则：**不要从仓库复制分发包，要从白名单生成分发包。**

只允许进入分发包：

```text
bin/ 或 bundle 后的入口
lib/ 或 bundle 后的运行代码
package.json
weilai.bat / 启动器
system.example.json
channels/*.example.json
后续的 core.node / license 模块
```

禁止进入分发包：

```text
.git/
docs/
test/
skill/
archive/
logs/
telemetry-out/
system.json
channels/*.json
secrets.json
_video_state.json
*.sig.json
*.submissions.jsonl
```

需要配套 `dist audit`：生成包后扫描敏感词和真实值，扫到即失败。

建议扫描：

```text
工程总报告
反逆向
PLAN
secrets
aavid
planId
sig.json
submissions
真实账户 ID
真实计划 ID
真实关键词
```

## 4. 主线 Phase 1：初期分发全量操作 skill

本轮最终裁决：初期分发时，仍然分发全量操作 skill。

原因很现实：如果 skill 不完整，客户侧 AI 很难稳定操控 CLI，反而会误用命令、漏掉 dry-run、搞错 paid 风险边界。对初版分发来说，“AI 能不能安全把工具用起来”比“完全隐藏操作流程”更重要。

因此这里接受一个明确代价：

```text
分发全量操作 skill
→ 客户/客户 AI 会知道 CLI 操作流程
→ 但能显著降低误操作和售后成本
```

允许分发 skill 写：

```text
ready -> sync -> delete -> md5fix -> upload -> reconcile
delete 默认 dry-run
paid 必须主管解锁
不要 taskkill Chrome
失败后如何重跑
```

仍然要从分发 skill 中删除或弱化：

```text
真实 aavid / planId / account / 路径 / 关键词
抓签名重放的具体接口与字段
NO_CHOOSER 的完整根因和探针过程
关键选择器/URL/请求体细节
内部源码文件逐行解释
历史翻车复盘和调试脚本说明
```

这不是“低泄漏”，而是“可用性优先的可控泄漏”。中长期再做产品化命令面，例如：

```text
start
stop
status
report
doctor
setup
```

等这些高层命令足够稳，再逐步把 skill 从“流程编排手册”降级成“产品操作手册”。现阶段不强行薄化。

## 5. 主线 Phase 2：离线授权占位

联网授权、自动充值、AI 网关先作为备选。因此主线先做离线授权接口边界，而不是完整商业计费。

工程目标：

```text
先把授权调用点埋好
以后可替换成本地 license、在线 lease、发卡兑换或 AI 网关
```

建议新增统一接口：

```js
await requireEntitlement({
  command,
  channel,
  feature
});
```

第一版可用本地 `license.json` 或 signed token：

```json
{
  "customer": "customer-a",
  "machine": "machine-hash",
  "expires": "2026-09-30",
  "features": ["free", "paid", "ai-report"]
}
```

注意：离线授权不能只做：

```js
if (!license.ok) exit
```

后续应让授权结果参与策略解密：没有有效 license，就解不开关键 URL、选择器、接口关键字和策略表。

## 6. 主线 Phase 3：Rust 小内核

Rust 不用等最后。确定边界后可以先做。

不建议包：

```text
CDP 点击
DOM 注入
file chooser 事件
ready/session 收敛
页面选择器交互
```

这些逻辑依赖浏览器运行态，搬进 Rust 会很痛，且运行时仍可被观察。

建议包：

```text
license 验签
policyKey 派生
核心策略解密
URL/选择器/接口关键字解密
worklist/status/delete/reconcile 的纯计算
platform rows 归一和筛选
```

形态：

```text
JS 做浏览器自动化的手脚
Rust 藏钥匙、策略和部分脑子
```

示例边界：

```js
import core from './core.node';

const policy = core.unlockPolicy(machineInfo, license);
const work = core.computeWorklists(state);
const targets = core.computeDeleteTargets(platform, state, channel);
```

## 7. 主线 Phase 4：JS 打包混淆

Rust 小内核跑通后，再做 JS 保护链。

推荐：

```text
esbuild 打包成单文件
→ javascript-obfuscator 强混淆
→ 字符串加密
→ 控制流混淆 / self-defending 按需开启
```

重点保护字符串：

```text
平台 URL
接口路径
选择器
中文按钮文案
CDP helper 字符串
fetch 重放逻辑
错误提示中的实现线索
```

混淆只能提高阅读成本，不能保护已经被运行时观察到的行为。因此混淆必须配合剔密、受控分发 skill、Rust policy 和运行时痕迹收敛。

## 8. 主线 Phase 5：SEA/exe 最后封箱

SEA/exe 是最后一步，不建议提前做。

原因：

```text
早做会增加调试成本
加载 .node 原生模块会更麻烦
SEA 单独防护不强
```

建议最终分发形态：

```text
weilai.exe
core.node
system.example.json
channels/*.example.json
license cache
```

`.node` 可以放在 exe 旁边，或启动时解压到临时目录再加载。这个问题等最后封装时解决。

## 9. 主线 Phase 6：运行时痕迹收敛

代码加密盖不住运行时行为，但可以减少“现成答案”。

分发版建议：

```text
monitor 默认隐藏或禁用
签名缓存短 TTL / 加密 / 退出清理
日志脱敏 aavid、planId、mid、完整 URL
Network.enable 用完尽量关闭
不长期落明文 postData / sig
错误信息不要暴露具体平台策略
```

尤其注意：签名缓存、遥测 JSONL、日志和台账旁车文件，泄密程度可能高于源码变量名。

## 10. AI 功能路线：主线暂缓，保留三档

AI 功能能提高售价，但如果没有后端，不能把供应方 API key 放进客户端。把 API key 包进 token 本质仍会泄钥。

因此 AI 先不进入加密主线，保留三档备选：

| 档位 | 做法 | 优点 | 问题 |
|---|---|---|---|
| 本地 BYOK | 客户填自己的 API key | 无后端、最快 | 不能赚 token 差价 |
| 轻量 AI 网关 | Cloudflare Worker/Vercel 转发 OpenAI/DeepSeek | 可隐藏 key、可计费 | 需要后端、额度账本 |
| 云端伪 agent | 云端 agent + 本地 worker | 最强产品化 | 工程量最大 |

当前建议：先做离线主线。AI 可先作为未来专业版/托管版能力，不阻塞加密分发。

## 11. 支付与充值路线：主线暂缓

自动扫码充值、每人独立额度、收入分流到自己钱包，本质需要：

```text
支付订单
支付回调
客户额度账本
AI 网关扣费
```

这一定需要你控制的服务端。支付宝/微信/发卡平台能简化收款，但不能替你维护 AI 额度和扣费逻辑。

保留三档备选：

| 档位 | 做法 | 适用阶段 |
|---|---|---|
| 手动收款/发卡 | 个人码或发卡平台，人工/半自动 | 验证客户愿不愿付钱 |
| 发卡平台 + 充值码 | 平台收钱并发码，本地或轻量接口兑换 | 小规模商业化 |
| 支付 webhook + AI 网关 | 自己维护订单、余额、扣费 | 正式托管 AI |

当前不建议把支付系统纳入近期主线。

## 12. 推荐实施顺序

### 近期必须做

```text
1. 写 make-dist 白名单脚本
2. 写 dist audit
3. 整理分发版全量操作 skill：保留操作流程，删真实配置、源码解释和内部调试细节
4. 设计产品化命令面草案：start/status/report/stop/doctor
5. 暂不强行隐藏内部命令；先保证 AI 可用，后续再收敛 help
```

### 中期做

```text
6. 增加 license 接口边界
7. 设计 policy 数据结构，把关键字符串/策略表从 JS 明文中抽出来
8. 做 Rust/N-API core.node
9. JS bundle + obfuscator + string encryption
```

### 最后做

```text
10. SEA/exe 封箱
11. 日志/签名缓存/遥测脱敏和清理
12. 分发包全量审计
```

### 暂缓备选

```text
AI BYOK
轻量 AI 网关
扫码支付/发卡充值
云端伪 agent
```

## 13. 最终判断

当前最稳路线不是“一步到位做云端/支付/agent”，而是先做一个可被 AI 稳定操控的离线分发包：

```text
客户能通过 skill 正确使用 CLI
但拿不到内部文档、真实配置、测试探针和源码级解释
```

也就是说：

```text
先剔除最高危资料
接受操作 skill 的流程泄漏
再藏关键字符串和策略
再封装运行形态
最后再考虑联网商业化与产品化命令面
```

这条路线工程量可控，且不会把 Weilai-01 过早拖进支付系统、云网关、agent 平台这类新战场。
