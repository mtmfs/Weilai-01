## 1. 产品是什么 — 双通道过审业务模型

**业务痛点**：千川投放前，视频素材要过平台**审核**。审核有概率被拒，反复试错会烧投放号的审核额度、也耗时。

**套利思路**：用两个成本不同的账户分工——

| 通道 | 账户 | 角色 | 模式 | 花钱 | 端口 | UI 形态 | 单件最多传 |
|---|---|---|---|---|---|---|---|
| **jie3** | 捷沅3 | test 测试端 | 推商品 | ❌ 免费 | 9222 | 抽屉 drawer | 7 次 |
| **jie6** | 捷沅6 | delivery 投放端 | 推直播间 | ✅ 真花钱 | 9223 | 创意 tab | 3 次 |

**流程**：本地视频 →（传到免费 jie3 跑过审）→ 过审则**封存** →（搬到付费 jie6 投放）→ jie6 也过审则**交付**。被拒的就**改 MD5**（ffmpeg 重封装，骗过平台去重）重传，每通道最多传 maxUploads 次；传满仍败则**作废**。

**两个关键手法**：
- **改 MD5 绕去重**：平台按文件哈希去重，同一被拒视频直接重传会被秒拒。`md5fix` 用 ffmpeg `-c copy`（不重编码）重封装 + 抹元数据 + 塞随机 UUID 注释，产出"内容相同、哈希不同"的新副本，让它被当成新素材重新审核。
- **双号分工**：免费号承担试错，付费号只接已验证过审的，省额度、控风险。

**视频生命周期**：`testing → sealed → delivering → delivered`（成功）或 `→ scrapped`（失败）。这套状态由台账机器维护（见 §3）。

## 2. 系统架构 — 分层与数据流

### 2.1 三层结构

```
横切层（正交·包裹所有业务·不占业务高度）
  cdp.mjs        CDP 原语：连调试Chrome标签、页内去重助手、抓签名/重放fetch、受信任手势
  session.mjs    上下文就绪：7探针+8幂等动作+ready收敛调度器（从任意页面收敛到"可上传"）
  guard.mjs      韧性：每个碰浏览器的动作前后 扫弹窗+断言身份，漂移/掉登录/签名冷 自愈一次
  telemetry+bandit  遥测：旁路被动录制 + 分时段过审率 + Thompson采样择时
══════════════════════════════════════════════════════════
业务叶子（原子动作）
  sync    拉平台审核归台账        delete  删过审+被拒副本腾槽
  md5fix  改哈希绕去重(纯本地)     upload/submit  注入+逐文件超时提交
  state   双通道台账状态机（机器真源）
══════════════════════════════════════════════════════════
编排器（组合叶子）
  prep = sync→delete→md5fix          test-round = ready+prep+upload（jie3一轮）
  cycle = 多轮{sync→delete→md5fix→upload}直到待传清零
```

### 2.3 单写者契约（根治计数错乱）

## 3. 数据模型 — 台账 schema · config · 状态机

### 3.1 台账 `_video_state.json`（机器真源，在 `H:\DD\6-18-魏-指纹\`）

```jsonc
{
  "channels": { "jie3": {role,account,aavid,mode,planId,funded,maxUploads}, "jie6": {…} },
  "pipeline": ["jie3", "jie6"],          // 流水线顺序：测试端 → 投放端
  "videos": {
    "<文件名>": {
      "name": "<文件名>",
      "ch": {
        "jie3": { "uploads":2, "passed":true, "last_status":1, "last_mid":"765…", "last_ts":178… },
        "jie6": { … }
      },
      "stage": "delivered",              // 由 recomputeStage 算出
      "scrap_reason": null
    }
  },
  "rounds": { "<token>": {ts,n} }        // 轮次幂等令牌（防30min内误重跑双计）
}
```

每通道字段：`uploads`(传了几次)、`passed`(过审,单调真)、`scrapped`(作废,单调真)、`last_status`(平台审核码：1过/2拒/3审核中)、`last_mid`(平台素材ID)、`last_ts`(本轮时间戳)。

> ⚠️ **schema 漂移注记**：老条目可能缺 `scrapped` 字段（早期 `blankCh()` 没有它）。例如实台账里 `…李傲奇-测试_8`（双通道 delivered）的 jie3 块就没有 `scrapped`，而 `…测试_73`（jie3 传满 5 次被拒）有 `scrapped:true`。读取代码对缺字段做了兜底，但分析台账时要知道这点。

### 3.2 config 三层（硬编码全外提，命令行只用 ASCII）

| 文件 | 内容 | 入库? |
|---|---|---|
| `system.json` | 机器/项目级：chrome 路径与 flags、ffmpeg 候选、台账路径、关键词`魏文彬`、超时、并发、批大小 | ✅ |
| `channels/{jie3,jie6}.json` | 通道级：account/aavid/advId/mode/planId/port/ui/funded/maxUploads | ✅ |
| `secrets.json` | 母账号 `{mother:{email,pwd}}` | ❌ gitignored，或用环境变量 `QC_MOTHER_EMAIL/PWD` |

### 3.3 stage 状态机

```
  上传到jie3 ─▶┌─────────┐  jie3传满7次仍last_status=2 ─▶ scrapped(终态·失败)
              │ testing │
              └────┬────┘
        jie3过审   ▼
              ┌────────┐
              │ sealed │ (jie3过审, 还没推jie6)
              └───┬────┘
       推jie6上传  ▼
            ┌────────────┐  jie6传满3次仍被拒 ─▶ scrapped
            │ delivering │
            └─────┬──────┘
       jie6过审    ▼
            ┌───────────┐
            │ delivered │ (终态·成功, passed 优先于 scrapped)
            └───────────┘
```

`recomputeStage`（`lib/state.mjs`）规则，按顺序：
1. 满 maxUploads 且最后被拒 → 该通道 `scrapped=true`（单调钉死）。
2. jie3 未过审 → `scrapped` 则 `scrapped`，否则 `testing`。
3. jie3 过审、jie6 没传 → `sealed`。
4. jie3 过审、jie6 过审 → `delivered`（过审优先于作废）。
5. jie3 过审、jie6 作废 → `scrapped`；否则 → `delivering`。

**worklist 派生**（驱动上传，`worklists()`）：
- `test_toupload`：jie3 一次没传的。
- `test_reupload`：jie3 **只取 `last_status===2`（被拒）**的——审核中(3)的在飞件不重传（`24f845b` 修复，旧报告曾误标"必修"，实已修，见 §7）。
- `sealed`/`deliv_toupload`：jie3 过审待推 jie6 的。

## 5. 关键设计决策日志（为何这么做）

> 这些"看着绕"的设计，背后都有踩过的坑。这是历史文档普遍缺的"连接组织"。

1. **session = 7 探针 + 8 幂等动作 + 收敛调度器**（而非一条直线脚本）。
   *为何*：冷启动是 bug 最密集的环节，人可能停在任意页面。探针判"我在哪步"、动作"已达成就跳过"、调度器"只补缺的步"，覆盖 6 种起始态。*代价*：复杂度高，但换来"从任意态自愈到就绪"。

2. **上传"逐文件超时"而非"整批门控"**。
   *为何*：旧逻辑等整批传完才提交，一个龟速件能逼已传完的 9 个陪等最长 45min。改成传完即可提交、龟速件超时踢回。*代价*：完成件进度条会移出列表导致 `done` 计数不可靠（Bug A），改用"确定按钮可点 + inprogSum/total 变化"判进度（`a6e727c` 已修）。

3. **NO_CHOOSER 真根因 = 后台布局节流，不是冷加载**。
   *为何*：上传框点了不弹文件选择框，困扰最久。三层 instrument 钉死真因：页面在后台时抽屉布局不落定，上传框卡到屏外（实测 cx=2454 > 视口 1904），坐标点空。解药 `Emulation.setFocusEmulationEnabled(true)` 让后台页按聚焦渲染→布局落定。**早期"组件冷加载慢 / C 盘满"的诊断全是误判**（`28cce98`/`88fb212` 根治）。*代价*：视口硬化只加在了 upload 路径，delete/ready 仍裸奔（见 §6 待改二）。

4. **bump 按"注入"记账 + 事后对账**（而非按"提交"）。
   *为何*：上传瞬间平台素材还按 ID 命名、拿不到可靠的"文件↔素材"映射，按提交记账会漏记真提交件→重传→平台重复件（更糟）。故保留"按注入记账"（过记方向可恢复）。*代价*：Bug B 台账虚高，需事后 `reconcile` 对账（见 §6 待改一）。

5. **双实例独立 profile**（jie3@9222 / jie6@9223）。
   *为何*：两通道共享浏览器 session 会被顶号，导致账户漂移（有钱的 jie6 被顶到 jie3）。独立 profile 从结构上根治。*代价*：两进程写同一台账会 lost-update，所以现在串行（飞轮 S2 需加台账锁）。

6. **台账原子写 + .bak + 解析失败拒绝覆盖**。
   *为何*：覆盖写不抗崩溃；解析失败若静默返回空台账，随后 saveState 会用空数据清零好台账。现改为临时文件→rename、自动备份、坏档大声失败。

## 8. 技术债与可持续性评估（决策核心）

### 8.1 诚实的技术债清单

- **强耦合 UI 自动化**：靠 CDP 驱动网页 DOM + 重放抓来的签名请求，**平台改版/选择器漂移即停**（guard 会大声退出，但需人修选择器）。
- **写死的目标**：通道 jie3/jie6、关键词 `魏文彬`、计划 ID 全钉死，换目标要改 config。
- **硬环境前提**：双调试 Chrome + 固定 profile + C:/H:/I: 磁盘余量 + ffmpeg 路径，任一缺失即崩。
- **jie6 半残**：能传不能删、冷登录断，付费通道未真正自动化闭环。
- **无自动化测试守网**：离线单测放在仓库外（不入 git）、混沌测试是一次性脚本；改代码无 CI 兜底。
- **Bug B / mid 捕获不全**：台账与数据统计要靠事后人工对账。

### 8.2 三条路线与取舍

| 路线 | 做什么 | 成本 | 触发条件 |
|---|---|---|---|
| **A 维持现状·小步修**（推荐起步） | 只修 §6 待改一(reconcile)+二(视口硬化)，jie3 免费筛过审日常用 | 低 | 只用 jie3、或小批量投放 |
| **B 加固投放通道** | 补 jie6 delete/sweep(待改三)+登录链(五)+视口硬化，让双通道真全自动 | 中-高（含在有钱号上调试登录的风险） | 要常态化、规模化 jie6 投放 |
| **C 结构性升级** | S2 飞轮（双实例并行+台账锁）/ S9 API-direct（弃 UI 自动化、直接复现签名推字节） | 大 | 批量上规模、把吞吐当核心指标 |

**建议**：先走 **A**（低成本止血，让日常稳定可用）。**B/C 由投放业务量决定**——若 jie6 只是偶尔投几十个，维持半残+人工补位即可；若要把它当生产线天天大批量跑，再投 B、规模再上则评估 C。**不建议在没有明确放量需求时投 C**（UI 自动化的脆弱性用工程量堆不平，API-direct 是另一个量级的 R&D）。
