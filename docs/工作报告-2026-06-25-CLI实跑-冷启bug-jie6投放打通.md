## 三、踩的坑 / 失误（照实记，不藏）

1. **监控器误判**：第一次上传 `NO_CHOOSER`，我**先怀疑是 monitor 干扰**(它 attach 上传标签)。关掉 monitor 重跑**仍 NO_CHOOSER** → 证明**不是 monitor**。真凶是冷启首开时序（见 §六）。我把误判当头号嫌疑写了出来，是错的，已纠正。
2. **59 一锅端 = 错方法**：第一轮我按「积压 59 个一次性全传」跑，被用户当场纠正——要的是**小并发(10个/轮)多轮**，不是一次铺满。且那 59 里 23 个是旧哈希重传件、大概率重拒、白烧额度。**已干净作废**(node 被杀时 Chrome keep-alive 死、传输停在 <100%，0 提交、0 素材、台账零改动)。
3. **把守卫说大了**：我曾称 scratchpad 驱动的「fail-loud 守卫」能防 45min 空转。**准确讲，①②修库之前它防不住**——`runUpload` 内部 `inject→submit` 顺序调用，注入空后 submit 先空转 45min 才返回、守卫那时才看到。修了①②(runUpload 注入0直接抛)之后，守卫才真生效。这点我夸大了，已纠正。
4. **单线程串行(非飞轮)**：全程一轮传完才 sync/delete/下一轮，投放还得等迭代让出实例。用户指出**理想应是双实例并行(一边投放、一边迭代)+ 后台刷 md5**——我退成串行是图省事规避台账竞态，不是对的架构（详见 §七）。

## 六、★ 冷启 NO_CHOOSER bug（已修失败模式，未修根因）

**现象**：冷启 Chrome 后**首次**上传，注入第1批 `NO_CHOOSER` ×6 → 注入0 → submit 对空面板空转到 45min → 返回 0、**不报错、退出码0**。

**根因**：千川上传框是点击时临时建 `<input type=file>` 的 React 组件。冷启首开时框已渲染(`findBox` 命中)，但点击处理器**还没挂载完** → `clickAt` 点空 → 不弹 `fileChooserOpened`。满盘 C: 挤压磁盘缓存 → 组件 JS 冷加载慢 → 绑定窗口 > 重试预算 → 输竞速。组件进缓存后秒绑。**铁证**：预热后同坐标 clickAt 一点即触发 chooser。

**与代码改动无关**：纯平台前端冷加载时序；inject 逻辑本身没问题（06-24 暖态验证过）。

**★无人值守真正危害 = 静默空转**：纯自动化通宵跑会变「跑一夜、0上传、无报错、无告警」——最坏的失败。

**修复状态**：
- 🟡 **①② 已修+验证**：失败模式从「静默45min空转·退出0」→「秒级 E_GESTURE·退出15·可接住」。**这修的是失败模式(静默→大声)，不是根因。**
- ⬜ **③④ 根因预防(warm-up)未做**：openUploadPanel 可交互探测 / ready 预热上传组件。做完冷启动也不犯。本程靠「暖 Chrome 不 close」运营绕过未触发。

## 七、★ 飞轮架构(S2)讨论纪要

**用户指出**：理想态不是单线程串行，而是**双实例并行飞轮**——9222 jie3 不停迭代(传下一批)、9223 jie6 同时投放上一批过审、后台 md5fix 总比上传快一步刷好下批哈希。三条线重叠跑(Little 定律：吞吐=瓶颈速率，非各阶段相加)。审核是最长杆，串行=等审核时空转，飞轮=等审核时在传/在投/在刷哈希。

**我退成串行的真约束**：两进程**共写同一份台账会丢更新**（后写者覆盖前写者；saveState 原子写只防写坏、不防 lost-update 竞态）。

**飞轮要做对的关键 = 单台账属主**：不是起两个独立 node，而是**一个编排进程**开**两条 CDP 连接(9222+9223)**，jie3-迭代 ∥ jie6-投放 当两个并发 async 任务，共享**一份内存台账**、存盘**串一把锁**（两边改不同字段 ch.jie3/ch.jie6，加 save 互斥即零竞态）；md5fix 挂后台 child_process 池往上传队列前面喂哈希。这就是 S2。

**务实分两步**：
- **后台 md5fix 现在就能并行**(安全，只写 I:、不碰台账/Chrome)；
- **jie3∥jie6 真并行**需搭「单编排双CDP + 台账存盘锁」(backlog 评 S2「大/重大」)。

**取舍**：对本批(19+30在飞)飞轮即时收益有限(用户也认)；属**架构投资**，跑大批量才真香。
