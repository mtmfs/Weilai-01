## 高优先（正确性 / 会复发）

### 一、`weilai reconcile <ch>` 命令 — 把 Bug B 的手动救法自动化
- **是什么**：上传时注入 N 个文件、只 M 个真提交（卡死件），`bump` 却按 N 记账 → 台账虚高（说传了实际没传），含把 uploads=4 的件冤记成满 5 次。
- **影响**：每次 `submit < inject` 都污染台账；无人值守时**悄悄累积、还可能误判作废**。本程靠手写脚本救（scratchpad·跨轮即清·逻辑没进 CLI）。
- **位置**：新建 `bin/cmds/reconcile.mjs` + `lib/state.mjs` 加一个 reconcile 函数。
- **修法**（已验证有效·把手动脚本固化）：本轮上传的件共享同一 `last_ts`（runUpload 同一 `now`）。grace 期（如 30min，让平台素材出现+名字解析）后，对 testing/delivering 中 `last_ts==该轮ts` 且**平台无 live 素材**的件 → un-bump（`uploads-1`、`last_status`→2/null、清 `last_mid/last_ts`）。
- **前置**：无。**工作量**：中。

### 二、视口硬化扩到 delete / ready（现只 upload 有）
- **是什么**：这次给 `runUpload` 加了 `setDeviceMetricsOverride+setFocusEmulationEnabled`（钉死视口·根治 NO_CHOOSER）。但 `delete` 和 `ready` 也按坐标点击、**没加**。
- **影响**：`delete` 的 `reopenDrawerJie3` 用 `Input.dispatchMouseEvent` 按坐标点搜索框、`ready` 的 `lockPlan` 同理 → **后台/窄窗口时一样会点屏外**（表现成"搜索没输入/锁计划失败"，NO_CHOOSER 同款根因）。这次没踩只是凑巧在前台宽窗口。
- **位置**：`lib/delete.mjs::runDelete`（连接初始化处）、`lib/session.mjs::ready`（cdp 初始化处）；或更彻底——提到 `lib/cdp.mjs::connect()` 让所有连接统一硬化。
- **修法**：连接后加 `Emulation.setDeviceMetricsOverride({width:1920,height:1080,deviceScaleFactor:1,mobile:false})` + `setFocusEmulationEnabled({enabled:true})`。
- **前置**：无。**工作量**：小。⚠️ 若提到 `connect()`，需回归测 sync/ready/delete 不被新视口影响选择器。

## 中（功能缺口）

### 三、jie6 delete / sweep（creative-tab）实现
- **是什么**：`runDelete` 对 jie6（`ui=creative-tab`）直接抛 `E_CONFIG`（"待实现"）。
- **影响**：jie6 腾槽删不了——**含本程造的 4 个重复素材**（超时残件 + 这轮一起提交的"8 个素材"），现在清不掉、只能等审核重拒。
- **位置**：`lib/delete.mjs::runDelete`（jie6 分支）。
- **修法**：参考 jie3 的 set-opt 抓+重放，但 creative-tab 无抽屉——预抓 `material/set-opt` 签名（1 次 UI 删）→ `fetch` 批量重放 `{optType:delete,LegoMids,UseLegoMid:true}`。
- **前置**：jie6 ready（已通）。**工作量**：中。

### 四、delete 单目标行 E_SELECTOR 兜底
- **是什么**：删除目标恰好 1 个时，抽屉搜索结果不渲染该行 → `uiDeleteCapture` 抓不到 set-opt 签名 → 抛 E_SELECTOR 停。
- **影响**：自治轮（cycle）里 delete 只 1 目标时会停（驱动正确地大声停了，但本可兜底续跑）。
- **位置**：`lib/delete.mjs::reopenDrawerJie3` / `uiDeleteCapture`。
- **修法**：搜索无结果时翻页/换搜索词重试；或单目标时改用预抓签名直接重放（不依赖 UI 删那次）。
- **前置**：无。**工作量**：小-中。

### 五、jie6 冷 profile 登录链修复（报告#2/#3/#5）
- **是什么**：jie6 冷 profile 登出后，`ready jie6` 的登录态探测不触发 `login()` → 握手未鉴权卡死。现全靠"暖 profile 不登出"绕过。
- **影响**：整条投放通道鲁棒性悬在此；profile 一旦登出就断。
- **位置**：`lib/session.mjs::probeLoginStatus` / `login` / ready 的登录核验段。
- **修法**：修登出态探测（`agent.oceanengine` 落地页/`redirect/ad` 在登出时不亮 `/login`，但 `business.oceanengine/login` 最终会出现——探测漏判）；登录组件含反自动化"我已阅读并同意"自定义勾选（`.account-center-agreement-check`）。
- **前置**：需在**登出的 jie6 profile** 上调试 `login()`——**有风险**（jie6=有钱号）。**工作量**：中-高。

### 六、mid 捕获补硬（喂饱 S1 增量同步 + 修 pass-rate 数据）
- **是什么**：上传后抓平台分配的 materialId 记进 `last_mid`。现按文件名匹配，但平台上传当下**按 ID 命名**（几小时后才解析回文件名）→ 只抓到一部分（jie3 12-19/批、**jie6=0**）。
- **影响**：① 没 last_mid 的件 S1 增量查不到 → 退全量 sync 兜底（不致命）② `submissions.jsonl` 记录不全 → **pass-rate 分时段统计偏、S5 bandit 喂不饱**。
- **位置**：`lib/upload.mjs::pickSubmittedMids` / `runUpload` 捕 mid 段。
- **修法**：抓 `bind-video-to-owner` 请求（提交时带 `{file_name, video_id}`）→ vid → 匹配 `playInfo.VideoId` → materialId。**这是张晟钰报告里验证过的可靠"文件名↔素材"映射**，绕开命名延迟。或：上传后延时(分钟级)重抓。
- **前置**：无。**工作量**：中。
