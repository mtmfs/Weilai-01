# Weilai-01 授权系统设计（离线非对称签名 · 分发版不可自签）

> 状态：设计稿 · 2026-07-02
> 目标读者：签发方（你，持私钥）+ 维护者（AI，改客户端代码）
> 关系：本文只定**授权逻辑层**；防客户篡改内置公钥/验签逻辑属**代码保护层**，见 `docs/加密路线报告.md`（bundle/混淆/SEA/Rust）。两层叠加才完整，缺一不可。

---

## 0. 目标 / 非目标

**目标**
- 只有**持签发私钥的你**能签出有效令牌；客户端（含 AI、含客户技术部）**无法自签**解锁 paid。
- 令牌可**绑机器**（一台一令牌，防复制扩散）、可设**权限**（features）与**有效期**。
- 你手里一个**一键签发脚本**：输入机器码 + 权限 + 天数 → 输出令牌。
- 最小改动接入现有 auth 系统（`wl1.*` 格式、`verifyAuthToken`、session、install/unlock 全不动，只改**公钥来源**）。

**非目标（本文不解决，靠代码保护层）**
- 防客户**改分发的 JS 源码**把内置公钥换成自己的公钥（纯 JS 可改 → 必须 bundle+混淆+SEA/Rust 才挡得住；见加密路线报告）。
- 联网吊销、在线许可服务器（本设计是纯离线）。

---

## 1. 威胁模型

| 对手 | 手段 | 本设计的防线 |
|---|---|---|
| 客户/主管/AI | 跑 `auth unlock`、设环境变量、造密钥自签 | **分发版只认内置公钥、私钥不在场 → 自签的令牌验不过** |
| 客户技术部 | 读分发 JS、找 env 后门 | 分发版**无 env 公钥后门**（内置非空即锁死 env）；命令可从 help 隐藏 |
| 有经验逆向 | 改分发 JS 里的内置公钥 / 短路验签 | ⚠ 授权层挡不住，靠**代码保护层**（Rust 验签内核 / SEA / 混淆） |

**当前（毛坯）为什么等于没锁**：`EMBEDDED_PUBLIC_KEYS = []` 空，`publicKeys()` 收 env 公钥 → 调用者可"自造密钥对 + 自签令牌 + 用 env 喂自己的公钥"**自签自验**。本设计根除这一条。

---

## 2. 核心原则（一句话）

> **私钥只在你手里（离线）；公钥硬编进分发版；分发版一旦内置了公钥就彻底不认 env 公钥。**

- 非对称（Ed25519）：验签只需公钥，公钥公开无所谓；签发需私钥，私钥你独占。
- 一套代码，两种形态由 `EMBEDDED_PUBLIC_KEYS` 是否为空自动切换：
  - **空** = 开发/测试版：回退 env 公钥（本机自签测试，方便 AI/开发）。
  - **非空** = 分发版：只认内置公钥、忽略 env（**不可自签**）。

## 3. 密钥模型与生成

- 算法：**Ed25519**（现有 `verifyAuthToken` 已用，零算法改动）。
- **私钥**：你在离线机器生成并保管（`wl-issuer-private.pem`）。**绝不入库、绝不交给 AI、绝不上分发机**。
- **公钥**：DER/SPKI base64 一串，交给维护者内置进 `EMBEDDED_PUBLIC_KEYS`。公钥泄漏无害。
- 可多把公钥并存（轮换/多签发人）：`EMBEDDED_PUBLIC_KEYS` 是数组，任一验过即可。

### 3.1 密钥生成脚本 `tools/wl-keygen.mjs`（你在离线机跑一次）

```js
// 生成签发密钥对：私钥留你手里，公钥内置进分发版。跑：node tools/wl-keygen.mjs ./keys
import { generateKeyPairSync } from 'node:crypto';
import { writeFileSync, mkdirSync } from 'node:fs';
import { join } from 'node:path';

const out = process.argv[2] || '.';
mkdirSync(out, { recursive: true });
const { publicKey, privateKey } = generateKeyPairSync('ed25519');

const privPem = privateKey.export({ type: 'pkcs8', format: 'pem' });
const pubDerB64 = publicKey.export({ type: 'spki', format: 'der' }).toString('base64');

writeFileSync(join(out, 'wl-issuer-private.pem'), privPem, { mode: 0o600 });
writeFileSync(join(out, 'wl-issuer-public.der.b64.txt'), pubDerB64);
console.log('私钥 →', join(out, 'wl-issuer-private.pem'), '（离线保管，绝不外泄/入库）');
console.log('公钥(DER b64) →', join(out, 'wl-issuer-public.der.b64.txt'), '（交维护者内置 EMBEDDED_PUBLIC_KEYS）');
console.log('\n内置用公钥串：\n' + pubDerB64);
```

## 4. 令牌结构（沿用现有 `wl1.*`，不改格式）

```
wl1.<payloadB64url>.<Ed25519签名B64url>
```

payload（JSON）字段：

| 字段 | 含义 | 0 号令牌取值 |
|---|---|---|
| `iss` | 签发方标识 | `"weilai-issuer"` |
| `aud` | 受众，固定 | `"weilai-01"`（`AUDIENCE`） |
| `sub` | 授权对象（人/机备注） | `"supervisor:机器A"` |
| `features` | 权限（`*`=全部；或 `paid.write` 等；`paid.delete` 单列） | `["*"]` |
| `secrets` | 允许解析的 BYOK secret（`*`/`openai.apiKey`） | `["*"]` |
| `machine` | 绑定机器码（省略=任意机器可用） | 目标机的 `localMachineHash()` |
| `nbf` | 生效时间（Unix 秒） | 签发时刻 |
| `exp` | 过期时间（Unix 秒，**必填**） | 按需，如 +365 天 |

> 权限判定沿用 `featureAllowed`（`*`/`supervisor`/`组.*`/精确）、`secretAllowed`。`delete-paid --apply` 要求 `paid.delete`（见 `bin/weilai.mjs::requiredSupervisorFeature`）——0 号令牌 `["*"]` 天然覆盖。
> 机器绑定沿用 `validateClaims`：`payload.machine && payload.machine !== localMachineHash()` → 拒。

## 5. 验证链改造（唯一的客户端代码改动 · 核心）

### 5.1 `lib/auth/token.mjs`：`EMBEDDED_PUBLIC_KEYS` + `publicKeys()`

```js
// 分发版：填入签发方公钥（DER SPKI base64，wl-keygen 产出）。可多把（轮换/多签发人）。
// 非空 ⟹ 只认内置、忽略 env（不可自签）；空 ⟹ 开发测试版回退 env。
const EMBEDDED_PUBLIC_KEYS = [
  // 'MCowBQYDK2VwAyEA....(你的公钥DER b64)....',
];

function publicKeys() {
  // ★分发版：内置非空 → 只认内置，彻底不看 env（根除自签后门）
  const embedded = [];
  for (const k of EMBEDDED_PUBLIC_KEYS) {
    try { embedded.push(createPublicKey({ key: Buffer.from(k, 'base64'), format: 'der', type: 'spki' })); }
    catch (e) { throw authErr('E_CONFIG', `内置公钥无法解析：${e.message}`); }
  }
  if (embedded.length) return embedded;   // 分发版：到此为止，env 一律忽略

  // ★仅当没有任何内置公钥（= 开发/测试构建）才回退 env，方便本机签测试令牌
  const keys = [];
  const der = process.env.WEILAI_LICENSE_PUBLIC_KEY_DER_B64;
  if (der) { try { keys.push(createPublicKey({ key: Buffer.from(der, 'base64'), format: 'der', type: 'spki' })); } catch (e) { throw authErr('E_CONFIG', `WEILAI_LICENSE_PUBLIC_KEY_DER_B64 无法解析：${e.message}`); } }
  const pem = process.env.WEILAI_LICENSE_PUBLIC_KEY;
  if (pem) { try { keys.push(createPublicKey(pem.replace(/\\n/g, '\n'))); } catch (e) { throw authErr('E_CONFIG', `WEILAI_LICENSE_PUBLIC_KEY 无法解析：${e.message}`); } }
  return keys;
}
```

> 现在 `EMBEDDED_PUBLIC_KEYS` 存的是 DER-b64 字符串（与 keygen 产出一致），故内置分支也用 `Buffer.from(k,'base64')` 解析——比原来直接 `createPublicKey(k)` 更明确。原代码把 EMBEDDED 当 PEM/KeyObject，本设计统一成 DER-b64 字符串。

**其余一律不动**：`parseToken`/`verifyAuthToken`/`validateClaims`/`installAuthToken`/`loadAuthToken`/`session.mjs`/`secrets.mjs`/CLI 的 `auth install-token|unlock|lock|status`。改动面 = 1 个函数 + 1 个常量。

### 5.2 分发构建校验（`dist audit`，防漏填/留后门）

分发打包时断言（脚本或 CI）：
1. `EMBEDDED_PUBLIC_KEYS` **非空**（否则分发版会退回 env、可自签）——空则**构建失败**。
2. 分发包内 grep 不到 `WEILAI_LICENSE_PUBLIC_KEY`（env 分支应随开发依赖被 bundle 剔除或压死）。
3. 与 `docs/加密路线报告.md` 的剔密白名单合并执行。

## 6. 签发侧：一键授权脚本 `tools/wl-issue.mjs`（你在离线机跑，**不入分发包**）

```js
// 一键签发令牌（需私钥）。示例：
//   node tools/wl-issue.mjs --key ./keys/wl-issuer-private.pem --machine <客户机器码> --sub "supervisor:机器A" --days 365
//   node tools/wl-issue.mjs --key ... --any --features paid.write,paid.read --days 90   # 不绑机器、限权限
import { readFileSync } from 'node:fs';
import { createPrivateKey, sign } from 'node:crypto';

function arg(name, def) { const i = process.argv.indexOf('--' + name); return i > -1 ? process.argv[i + 1] : def; }
const has = (name) => process.argv.includes('--' + name);

const keyPath = arg('key');
if (!keyPath) { console.error('必须 --key <私钥pem>'); process.exit(2); }
const priv = createPrivateKey(readFileSync(keyPath, 'utf8'));

const days = Number(arg('days', '365'));
const now = Math.floor(Date.now() / 1000);
const featuresCsv = arg('features', '*');
const secretsCsv = arg('secrets', '*');

const payload = {
  iss: arg('iss', 'weilai-issuer'),
  aud: 'weilai-01',
  sub: arg('sub', 'supervisor'),
  features: featuresCsv.split(',').map(s => s.trim()).filter(Boolean),
  secrets: secretsCsv.split(',').map(s => s.trim()).filter(Boolean),
  nbf: now - 60,
  exp: now + days * 86400,
};
// 绑机器：默认必须 --machine <码>；显式 --any 才签"任意机器可用"（风险高，慎用）
const machine = arg('machine');
if (machine) payload.machine = machine;
else if (!has('any')) { console.error('必须 --machine <客户机器码>，或显式 --any 签任意机器令牌'); process.exit(2); }

const pb = Buffer.from(JSON.stringify(payload)).toString('base64url');
const sig = sign(null, Buffer.from(pb), priv).toString('base64url');
const token = 'wl1.' + pb + '.' + sig;

console.log('sub      :', payload.sub);
console.log('features :', JSON.stringify(payload.features), '| secrets:', JSON.stringify(payload.secrets));
console.log('machine  :', payload.machine || '(任意机器)');
console.log('有效期   :', new Date(payload.exp * 1000).toISOString());
console.log('\n令牌（交客户 auth install-token）：\n' + token);
```

**0 号令牌** = 最高权限、绑目标机、长期：
```
node tools/wl-issue.mjs --key ./keys/wl-issuer-private.pem --machine <目标机器码> --sub "root:机器A" --features "*" --secrets "*" --days 3650
```

## 7. 客户激活流程（端到端）

```
① 客户机跑：weilai auth status            → 读出"本机码"(localMachineHash 的 32 hex)
② 客户把本机码发给你
③ 你在离线机：node tools/wl-issue.mjs --key 私钥 --machine <本机码> --sub ... --days ...  → 得令牌
④ 你把令牌发给客户
⑤ 客户机：weilai auth install-token <令牌>      → 写 license.json（内置公钥验签通过才收）
⑥ 需要 paid 时：weilai auth unlock [--all-day]  → 建 session（120min / 24h）
⑦ 跑 upload-paid / run-paid / run-both …
```

- 令牌绑机器 → 换机无效（防复制）；换机重走 ①–⑤。
- 分发版**无 env 公钥后门** → 客户/AI 自签的令牌在第 ⑤ 步 `verifyAuthToken` 直接拒（签名验不过内置公钥）。

## 9. 安全边界与运维

- **私钥保管**：离线、加密存储、备份；泄漏 = 全线失守（等于交出签发权）。轮换：生成新密钥对 → 新公钥**追加**进 `EMBEDDED_PUBLIC_KEYS`（旧的留一段过渡期再删）。
- **绑机器**：默认必绑（`wl-issue` 不给 `--machine` 且无 `--any` 直接拒），防一个令牌传遍全公司。
- **有效期**：`exp` 必填；0 号可长（3650 天），普通客户建议短（90–365 天）到期续签＝一种"软吊销"。
- **吊销**：纯离线无在线吊销；靠"短有效期 + 到期不续"实现。要硬吊销需联网许可服务（超出本设计，见加密路线报告的联网备选）。
- **机器码**：`localMachineHash = sha256(platform|hostname|username).slice(0,32)`（`shared.mjs`）——注意它随主机名/用户名变；客户换名/换用户令牌失效（可接受，重签即可）。
- **令牌泄漏**：绑机器 + 有限期把危害限制在"该机、到期前"。

## 10. 落地步骤（谁做什么）

1. **你**：离线机 `node tools/wl-keygen.mjs ./keys` → 得私钥（自己收）+ 公钥串。**只把公钥串给我。**
2. **我**：把公钥填进 `EMBEDDED_PUBLIC_KEYS` + 按 §5.1 改 `publicKeys()` + 新增两个 `tools/` 脚本 + 加 dist-audit 断言；跑离线测试确认（开发机 EMBEDDED 若填了则本地 env 测试令牌失效，测试改用你签的令牌或临时空 EMBEDDED 分支跑）。
3. **你**：`node tools/wl-issue.mjs --key 私钥 --machine <本机码> --features "*" --days 3650` 签 0 号令牌 → `auth install-token` → `auth unlock`。
4. 验证：**不带任何 env、用你签的令牌能解锁；随便自造一个令牌（或改 env）一律被拒**。

> 注意：第 2 步一旦把你的公钥填进 `EMBEDDED_PUBLIC_KEYS`，**本机开发/CI 里我之前用的 env 测试令牌就自动失效了**（分发逻辑忽略 env）。测试有两种走法：(a) 测试专用构建保持 EMBEDDED 空、走 env；(b) 用你签的一个"测试专用令牌"。建议 (a)：`cli-behavior-test`/`auth-test` 现在正是靠 env 注入公钥自签测试的，保持"测试时 EMBEDDED 空"即可继续绿，真分发包才填公钥。

## 11. 扩展：令牌内嵌临时 API key（与上文正交，可叠加）

需求：把供应方 API key 随令牌下发，客户端不配 BYOK env 也能用 AI 功能。**与"私钥自控/分发不可自签"完全正交**——那套保护"谁能签令牌"，本节关乎"key 的机密性"，两件事。

### 11.1 必须先认清：令牌签名 ≠ 加密

`wl1.<payloadB64url>.<sig>` 的 payload 是 **base64url 编码、不是加密**。签名只保证"未篡改 + 是你私钥签的"，**不保护 payload 机密性**：任何拿到令牌的人 `base64 -d` 即可读出 payload 全部内容。**所以直接把明文 key 放 payload = 明文泄漏**（这正是 `docs/加密路线报告.md`"包进 token 仍泄钥"的含义）。

### 11.2 三条路（按 key 价值选）

| 方案 | 机密性 | 复杂度 | 适用 |
|---|---|---|---|
| **A 明文嵌入** | 无（base64 解码即见） | 最低 | 临时、低价值、可随时轮换的 key + 绑机 + 短期 |
| **B 令牌内密文 + 客户端内置对称密钥解密** | 抬高成本（离线下内置密钥终可被抠出解密） | 中（+ 加解密 + 藏密钥靠代码保护层） | 想挡"随手 base64 看到"，接受"专业逆向仍可提取" |
| **C 联网网关** | 真不泄（key 只在你服务器） | 高（需后端） | 高价值/长期 key（超出纯离线，见加密路线报告联网备选） |

> 离线铁律：**任何客户端能用的 key，客户端持有者最终都能提取**。A/B 只把"提取成本"从 0 抬到"需逆向"，C 才是真不泄。临时功能建议 A 或 B，务必配**绑机器 + 短 exp（如 3–7 天）+ 用完轮换**，把泄漏窗口压到最小。

### 11.3 实现（A/B 通用骨架）

payload 加字段承载 key：
- 方案 A：`"apiKeys": { "openai.apiKey": "sk-..." }`（明文）。
- 方案 B：`"encKeys": { "openai.apiKey": "<AES-256-GCM 密文b64: iv|tag|ct>" }`，客户端用内置对称密钥 `EMBEDDED_KEY_ENC`（与公钥同受代码保护层保护）解密。

`lib/auth/secrets.mjs::resolveSecret` 改为**三级取值**（令牌优先、BYOK 兜底，授权门控不变）：
```
1. 当前 install 的令牌 payload：apiKeys[name]（A）或 decrypt(encKeys[name])（B）→ 有则用；
2. 回退 BYOK 环境变量（现状不变）；
3. 都没有 → E_CONFIG。
仍需 secretAllowed(payload.secrets, name) 通过；resolve --json 仍只回 masked、不回明文。
```
`tools/wl-issue.mjs` 加 `--api-openai <key>`（A 写明文进 apiKeys；B 用内置对称密钥加密后写 encKeys）。

### 11.4 落地增量（叠加在 §8 上）

| 文件 | 增量 |
|---|---|
| `lib/auth/token.mjs` | （方案 B）加 `EMBEDDED_KEY_ENC` 内置对称密钥 + 解密辅助；payload 容许 `apiKeys`/`encKeys`（随 payload 一起被签名保护完整性，验签逻辑不变） |
| `lib/auth/secrets.mjs` | `resolveSecret` 三级取值（令牌 → BYOK env → E_CONFIG） |
| `tools/wl-issue.mjs` | `--api-openai <key>`（A 明文 / B 加密写入 payload） |

> 令牌一旦内含 key（A/B），**令牌本身即"含密材料"**——签发日志/聊天/传输都按密钥对待，别落公开渠道；配短 exp + 绑机 + 轮换。
