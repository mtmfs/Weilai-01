## 3. 关键发现 / 认知校正

1. **平台单次导入有上限 ~10（操作者指出，代码佐证）**：超额会弹"仅前N个/超过/最多"对话框丢弃多的（`upload.mjs` inject 已处理该弹窗）。**分批是平台逼的，不是客户端瞎分。**
2. **`NO_CHOOSER` 的真因分两类**：①"框卡屏外"——视口硬化已基本根治（`findBox` 只返回视口内坐标）；②"同一会话里背靠背做第二次导入"——第二次点框不弹 chooser，这是**当前实跑撞上的**。一坏批 `break` 还连累后续批。
3. **正确迭代姿势 = 飞轮 + 每 tick 只导一批（`--batch ≤10`）**：等于把"9+9"拆成两个干净独立会话，天然避开第二次导入。飞轮 `hasLocal` 还自动滤掉无源陈旧件 + 返工件强制新哈希。
4. **无"等审核"死等环节**：飞轮连续 harvest（扫审核）+ release（边改边传），uploads 与审核并行 pipeline；唯一延迟是平台自身审核耗时，且被其它上传盖住。
5. **`delete` 的 `E_SELECTOR` 是偶发**（单目标行）：本次 tick3 自愈、成功删 11 个腾槽（`status 200 success`），非常态故障。
6. **陈旧件"假上传"机制坐实**：`upload` 取副本判据查的是 `I:\md5fix\` **产物**（非 flatRoot **源**），与 `md5fix` 查源不一致 → 源已删但旧副本还在时会传过期内容。
